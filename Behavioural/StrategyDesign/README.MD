> üéØ **Strategy Design Pattern**

---

## üß† What Is It?

> Strategy Pattern lets you define **a family of algorithms**, put them in separate classes, and make them **interchangeable at runtime**.

You separate the **"what to do"** from the **"how to do it"** ‚Äî and inject different behaviors as needed.

---

## üì¶ Real-Life Analogy

Imagine a **navigation app**:

* You can choose between:

  * üöó Car route
  * üö≤ Bike route
  * üö∂ Walk route

All these are **strategies** for "navigating" ‚Äî the app uses one depending on context.

---

## ‚úÖ Java Structure

### 1. Strategy Interface

```java
interface PaymentStrategy {
    void pay(int amount);
}
```

### 2. Concrete Strategies

```java
class CreditCardPayment implements PaymentStrategy {
    public void pay(int amount) {
        System.out.println("Paid " + amount + " with Credit Card");
    }
}

class PayPalPayment implements PaymentStrategy {
    public void pay(int amount) {
        System.out.println("Paid " + amount + " with PayPal");
    }
}
```

### 3. Context (uses the strategy)

```java
class ShoppingCart {
    private PaymentStrategy paymentStrategy;

    public void setPaymentStrategy(PaymentStrategy strategy) {
        this.paymentStrategy = strategy;
    }

    public void checkout(int amount) {
        paymentStrategy.pay(amount);
    }
}
```

### 4. Client Code (uses context)

```java
public class Main {
    public static void main(String[] args) {
        ShoppingCart cart = new ShoppingCart();

        cart.setPaymentStrategy(new CreditCardPayment());
        cart.checkout(500);  // Paid 500 with Credit Card

        cart.setPaymentStrategy(new PayPalPayment());
        cart.checkout(300);  // Paid 300 with PayPal
    }
}
```

## üí° Key Characteristics

| Feature                     | Explanation                                            |
| --------------------------- | ------------------------------------------------------ |
| ‚úÖ Replaces `if-else` blocks | You don‚Äôt write `if (payment == ...)`                  |
| ‚úÖ Runtime flexibility       | You can switch strategy dynamically                    |
| ‚úÖ Open/Closed principle     | Easily add new strategies without touching the context |

---

## üß® When NOT to Use

* If the behavior **never changes**
* If the variation is only **one or two lines**
* If it's **not reused**

---

## ‚úÖ Use Strategy Pattern When:

* You have **many related classes** differing only in **behavior**
* You want to **switch behaviors dynamically**
* You want to **encapsulate algorithms**

### üí≥ 1. **Payment Method (Classic)**

```java
interface PaymentStrategy {
    void pay(int amount);
}

class UPI implements PaymentStrategy {
    public void pay(int amount) {
        System.out.println("Paying ‚Çπ" + amount + " via UPI");
    }
}

class CreditCard implements PaymentStrategy {
    public void pay(int amount) {
        System.out.println("Paying ‚Çπ" + amount + " via Credit Card");
    }
}

class Checkout {
    private PaymentStrategy strategy;
    public Checkout(PaymentStrategy strategy) {
        this.strategy = strategy;
    }
    public void makePayment(int amount) {
        strategy.pay(amount);
    }
}
```

‚úÖ Inject strategy:

```java
new Checkout(new UPI()).makePayment(300);
new Checkout(new CreditCard()).makePayment(1000);
```

### üß† 3. **AI Behavior (in games)**

```java
interface EnemyStrategy {
    void move();

class Aggressive implements EnemyStrategy {
    public void move() {

```

‚úÖ Use:


### üïπ 5. **Travel Mode (Maps App)**


‚úÖ Use:


## üí° Key Things to Notice

* Each **strategy has its own class**
* The context (like `Navigator`, `Compressor`, etc.) is **decoupled** from specific logic
* Strategies can be **switched at runtime** or passed from config/injector
